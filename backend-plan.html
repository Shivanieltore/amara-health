Data storage strategy, database schema, and technology recommendations — June 2026

Section 1

User Data Collected from the Website

From "Get Started / Book a Demo" Forms

  • Full Name — Doctor's full name (required)
  • Medical Email — Work/clinic email address (required)
  • Clinic / Hospital Name — Practice or institution name (required)
  • Specialty — e.g. Cardiology, Endocrinology (required)
  • Role — Physician, Nurse Practitioner, Hospital Admin, IT Lead, Other
  • Product Interest — Amara Lite / DocDash / Both / Undecided
  • Pathway of Interest — Amara Lite / DocDash Layer / Not Sure Yet
  • Clinical Context / Requests — Free-text notes (optional)

From the Interactive Platform Demo Modal

  • Full Name — Doctor's name
  • Medical Email — Clinic email
  • Clinic / Hospital Name
  • Specialty
  • Role — Dropdown selection
  • Pathway of Interest — Toggle button selection
  • Clinical Context — Optional textarea

Session & Behavioural Tracking Data

  • Source — Which form submitted (demo_modal, get_started)
  • Pathway viewed — Amara Lite or DocDash tab active
  • Patient demo viewed — Which patient was selected
  • Tab section active — Health Trends, Medicines, Reports, HL7
  • Submission timestamp — UTC datetime
  • IP address / Country — For compliance and fraud prevention
Section 2

Recommended Database

Recommended
PostgreSQL via Supabase

Best fit for this structured lead capture use case

  • Structured relational data fits perfectly — all form fields are fixed columns
  • Supabase gives a REST API + authentication out of the box — no backend server needed initially
  • Free tier supports up to 500MB storage and 2GB data transfer per month
  • Built-in Row Level Security (RLS) for HIPAA-grade access control
  • Easy to extend with additional tables (appointments, patients, analytics)

Alternative Options

Alternative
Firebase Firestore
NoSQL document store. Good if form fields may vary per user type. Faster initial setup. Google Cloud ecosystem.
Alternative
MongoDB Atlas
Flexible schema if clinical notes or requirements expand significantly. Good for unstructured data alongside structured fields.
Section 3

Database Schema (PostgreSQL)

leads table

CREATE TABLE leads (
  id            UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  name          TEXT         NOT NULL,
  email         TEXT         NOT NULL,
  clinic        TEXT,
  specialty     TEXT,
  role          TEXT,
  pathway       TEXT,        -- 'lite' | 'docdash' | 'both' | 'unsure'
  notes         TEXT,
  source        TEXT,        -- 'demo_modal' | 'get_started'
  ip_address    INET,
  created_at    TIMESTAMPTZ  DEFAULT NOW()
);

demo_interactions table (optional analytics)

CREATE TABLE demo_interactions (
  id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id     TEXT,
  pathway_viewed TEXT,        -- 'lite' | 'dash'
  patient_id     TEXT,        -- 'pat-1' | 'pat-2' | 'pat-3'
  tab_active     TEXT,        -- 'health' | 'meds' | 'reports'
  created_at     TIMESTAMPTZ  DEFAULT NOW()
);

Section 5

HIPAA Compliance Notes

Since Amara Health handles clinical contact data, keep these in mind:

  • Do not store patient health data on the marketing website database — only lead/contact info
  • Enable Row Level Security (RLS) in Supabase so only authenticated admins can read lead data
  • Use HTTPS everywhere — Supabase enforces TLS by default
  • Sign a BAA with Supabase before storing any PHI — Supabase supports BAA on paid plans
  • IP addresses should be stored only if disclosed in your privacy policy
  • Audit logs — Supabase provides built-in audit logging on Pro plan
